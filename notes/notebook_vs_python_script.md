# 학습엔 노트북, 제작엔 파이썬 스크립트

> 의문: "에이전트는 결국 파이썬 코드(.py)로 만드는 거 아닌가? 그런데 왜 학습은 노트북(.ipynb)으로 하지?"

핵심: **최종 결과물(돌아가는 에이전트)은 `.py`로 만든다. 노트북은 거기까지 가는 학습/탐색 단계의 도구**다. 둘은 용도가 다르고, 실무에서도 함께 쓰인다.

## 노트북 vs 파이썬 스크립트

| | Jupyter 노트북 (`.ipynb`) | 파이썬 스크립트 (`.py`) |
|---|---|---|
| 실행 단위 | **셀 단위** (한 조각씩) | 파일 전체 한 번에 |
| 상태 유지 | 커널이 변수/객체를 **메모리에 계속 들고 있음** | 실행이 끝나면 사라짐 |
| 결과 확인 | 셀 바로 밑에 즉시 표시 (표, 그래프 그림, 메시지) | `print` / 로그로만 |
| 적합한 일 | **개념 실험, 탐색, 시각화, 학습** | **실제 애플리케이션, 배포, 재사용** |

## 왜 *학습 단계*에서 노트북이 유리한가

LangGraph 학습은 "한 줄 고치고 → 결과 즉시 보고 → 또 고치고"의 반복인데, 노트북이 이 루프에 강하다.

1. **한 번 만든 객체를 여러 번 재실험**
   `graph = builder.compile()` 셀을 한 번 실행해두면, 아래 셀에서 입력만 바꿔가며 `invoke`를 10번 돌려도 매번 처음부터 다시 안 돌려도 된다. `.py`라면 매 실행이 전체 재실행.
2. **그래프 그림이 셀 밑에 인라인으로 뜸**
   `draw_mermaid_png()` 결과 이미지가 바로 보인다. State / Node / Edge 구조를 눈으로 확인하며 익히기 좋다.
3. **스트리밍 중간 결과를 단계별로 관찰**
   `stream`이 뱉는 chunk를 한 단계씩 눈으로 따라갈 수 있다.
4. **설명 + 코드 + 결과가 한 파일에 남음**
   markdown 설명, 실행 코드, 그 출력이 같이 보존돼 나중에 다시 봐도 맥락이 살아 있다. = 학습 노트로 최적.

## 왜 *실제 제작*은 `.py`로 가는가

- **재사용** — 모듈로 쪼개 `from my_agent import graph` 처럼 가져다 쓴다.
- **배포** — FastAPI 같은 서버에 얹어 서비스로 만든다.
- **재현성** — import/실행 순서가 강제돼 항상 같은 결과.
- **협업/품질** — git diff·테스트·CI에 친화적. (노트북은 JSON이라 diff가 지저분하다.)

## 일반적인 흐름

```
노트북에서 개념 익히고 실험
        ↓ (검증되면)
   .py 로 옮겨 모듈/패키지화
        ↓
     서버에 배포
```

## 이 레포에서의 적용 방침

- **개념 학습 단계** (예: `frameworks/langgraph/basics/`) → **노트북**.
  그래프 메커니즘 자체를 셀 단위로 실험하며 익힌다.
- **실제 에이전트 제작 단계** (LLM을 붙이는 코드) → **`.py` 비중 ↑**.
  모듈화·재사용·배포가 목적.
- **이론/개요** → `.md` 로 분리 (예: `00_why_langgraph.md`).

한 줄 요약: **노트북은 배우려고, `.py`는 만들려고.**
